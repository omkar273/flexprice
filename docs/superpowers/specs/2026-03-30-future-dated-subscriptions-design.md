# Future-Dated Subscription Creation — Design Spec

**Date:** 2026-03-30
**Status:** Approved
**Scope:** Surgical (3 targeted fixes)

---

## Overview

Enable subscriptions to be created with a `start_date` in the future (pre-scheduling use case — customer signs a contract today, billing begins on a future date such as April 1). Simultaneously fix two related correctness gaps in the cancellation and billing layers that would cause ghost billing.

### Primary Use Case

A sales team closes a deal today. The contract start date is next month. The subscription should be created now (status: `active`), but metering and billing only begin at `start_date`. The first invoice is generated by the existing billing cron when `start_date` is reached — no special-case invoice logic on creation.

---

## Design Decisions

| Question | Decision |
|---|---|
| Subscription status during future period | `active` immediately — no new status needed |
| First invoice timing | Existing billing cron handles it naturally at `start_date` |
| Advance charges | Deferred to existing `BillingCycleAnchor` / cron logic |
| Cancellation fix scope | Forward-only — no backfill of existing cancelled subscriptions |
| Architecture | Surgical (3 targeted fixes) |

---

## Fix 1: Remove Future-Date Validation

**File:** `internal/api/dto/subscription.go`
**Lines:** 731–738

### Change

Remove this block from `CreateSubscriptionRequest.Validate()`:

```go
// DELETE THIS BLOCK
if r.StartDate != nil && r.StartDate.After(time.Now().UTC()) {
    return ierr.NewError("start_date cannot be in the future").
        WithHint("Start date must be in the past or present").
        WithReportableDetails(map[string]interface{}{
            "start_date": *r.StartDate,
        }).
        Mark(ierr.ErrValidation)
}
```

### Behavior After Change

- `start_date` accepts any date — past, present, or future
- Omitting `start_date` still defaults to `time.Now().UTC()` (unchanged)
- Subscription is created with status `active` regardless of `start_date`
- Events timestamped before `start_date` are already rejected by `isSubscriptionValidForEvent()` — no additional guard needed
- Billing cron naturally skips the subscription until `start_date` is reached (period start = `start_date`)

No changes required to the service layer or domain model.

## Fix 2: Propagate `EndDate` to Plan Line Items on Cancellation

**File:** `internal/service/subscription.go`

### Problem

`cancelAddonsForSubscription()` correctly sets `EndDate` on addon line items, but plan-level line items (those belonging to the subscription's plan) have no code path that sets their `EndDate` on cancellation. This leaves them open indefinitely, enabling ghost billing after cancellation.

### Change

Add a new helper function `cancelPlanLineItemsForSubscription(ctx, sub, effectiveDate)` called after `updateSubscriptionForCancellation()` sets the subscription's effective end date.

**Helper logic:**

1. Load all plan line items for the subscription
2. For each line item where `item.EndDate.IsZero() || item.EndDate.After(effectiveDate)`:
   - Set `item.EndDate = effectiveDate`
   - Persist the update via repository
3. Skip line items whose `EndDate` is already at or before `effectiveDate` — they are already correctly terminated

**Called for both cancellation types:**
- **Immediate cancellation** — `effectiveDate = now` (or provided date)
- **Scheduled cancellation** — `effectiveDate = cancel_at` (future date)

**Scope boundary:** Only plan line items. Addon line items continue to be handled by the existing `cancelAddonsForSubscription()`.

## Fix 3: Billing Respects Line Item `EndDate`

**File:** `internal/service/billing.go`

### Problem

`CalculateFixedCharges()` skips line items where `item.StartDate.After(periodEnd)` but does not skip items where `item.EndDate` has already passed. After Fix 2 ensures `EndDate` is always set on cancellation, this check is needed to enforce it at invoice time.

### Change

In `FilterLineItemsToBeInvoiced()` and `CalculateFixedCharges()`, add:

```go
// Skip line items that ended before the billing period started
if !item.EndDate.IsZero() && !item.EndDate.After(periodStart) {
    continue
}
```

### Behavior After Change

- A line item with `EndDate = March 31` is excluded from the April billing period
- Line items with `EndDate = zero` (no expiry) continue to be included normally
- Combines with Fix 2 to form the complete ghost-billing prevention: Fix 2 sets `EndDate` on cancellation; Fix 3 enforces it in billing

## Event Processing — No Changes Required

Event processing in `internal/service/event_post_processing.go` already correctly uses `event.Timestamp` for all resolution:

- `isSubscriptionValidForEvent()` checks `event.Timestamp` against `sub.StartDate` and `sub.EndDate`
- `item.IsActive(event.Timestamp)` checks the line item's `StartDate` and `EndDate` against the event timestamp
- Events before `start_date` are rejected for future-dated subscriptions automatically

No changes needed.

## Success Criteria

- A subscription with `start_date = future` can be created via the API without error
- Events timestamped before `start_date` are rejected (existing behavior, unchanged)
- Cancelling a subscription sets `EndDate` on all plan line items (not just addons)
- Billing does not invoice line items whose `EndDate` is before the billing period start
- No double billing across subscription boundaries
- No changes to existing subscriptions (forward-only)
