# Future-Dated Subscription Creation ŌĆö Design Spec

**Date:** 2026-03-30
**Status:** Approved
**Scope:** Surgical (3 targeted fixes)

---

## Overview

Enable subscriptions to be created with a `start_date` in the future (pre-scheduling use case ŌĆö customer signs a contract today, billing begins on a future date such as April 1). Simultaneously fix two related correctness gaps in the cancellation and billing layers that cause ghost billing.

### Primary Use Case

A sales team closes a deal today. The contract start date is next month. The subscription is created now (status: `active`), but metering and billing only begin at `start_date`. The first invoice is generated by the existing billing cron when `start_date` is reached ŌĆö no special-case invoice logic on creation.

---

## Design Decisions

| Question | Decision |
|---|---|
| Subscription status during future period | `active` immediately ŌĆö no new status needed |
| First invoice timing | Existing billing cron handles it naturally at `start_date` |
| Advance charges | Deferred to existing `BillingCycleAnchor` / cron logic |
| Cancellation fix scope | Forward-only ŌĆö no backfill of existing cancelled subscriptions |
| Architecture | Surgical (3 targeted fixes) |

## Fix 1: Remove Future-Date Validation

**File:** `internal/api/dto/subscription.go`
**Function:** `CreateSubscriptionRequest.Validate()`

### Change

Remove this block:

```go
// DELETE THIS BLOCK
if r.StartDate != nil && r.StartDate.After(time.Now().UTC()) {
    return ierr.NewError("start_date cannot be in the future").
        WithHint("Start date must be in the past or present").
        WithReportableDetails(map[string]interface{}{
            "start_date": *r.StartDate,
        }).
        Mark(ierr.ErrValidation)
}
```

### Invariant: StartDate Is Always Non-Nil After Default Assignment

Early in `Validate()`, `StartDate` is assigned a default before any checks run:

```go
if r.StartDate == nil {
    now := time.Now().UTC()
    r.StartDate = &now
}
```

The trial-date validations further down dereference `*r.StartDate` directly. This is safe because `StartDate` is always non-nil after the default assignment ŌĆö removing the future-date validation block does not break this invariant. No nil guard is needed, but this dependency must not be accidentally reordered.

### Behavior After Change

- `start_date` accepts any date ŌĆö past, present, or future
- Omitting `start_date` still defaults to `time.Now().UTC()` (unchanged)
- Subscription is created with status `active` regardless of `start_date`
- Events timestamped before `start_date` are already rejected by `isSubscriptionValidForEvent()` ŌĆö no additional guard needed
- Billing cron naturally skips the subscription until `start_date` is reached

No changes required to the service layer or domain model.

## Fix 2: Propagate `EndDate` to Plan Line Items on Cancellation

**File:** `internal/service/subscription.go`
**Functions:** `CancelSubscription()` and its helpers

### Problem

`cancelAddonsForSubscription()` correctly sets `EndDate` on addon line items, but plan-level line items have no code path that sets their `EndDate` on cancellation. This leaves them open indefinitely, enabling ghost billing after cancellation.

Note: the scheduled cancellation executor (`MarkCancellationScheduleAsExecuted` path) also does not call `cancelAddonsForSubscription()` ŌĆö that is a separate, pre-existing gap. Fix 2 does not address the executor path; see the Scheduled Cancellation Timing section below.

### Change

Add a new helper function `cancelPlanLineItemsForSubscription(ctx, sub, effectiveDate)` called from `CancelSubscription()` at the same point where `cancelAddonsForSubscription()` is called.

**Helper logic:**

```
filter := NewNoLimitSubscriptionLineItemFilter()
filter.SubscriptionIDs = []string{sub.ID}
filter.EntityType = lo.ToPtr(SubscriptionLineItemEntityTypePlan)  // plan items only
lineItems = SubscriptionLineItemRepo.List(ctx, filter)

for each line item:
    if item.StartDate.After(effectiveDate):
        skip  // item never became active; subscription-level EndDate protects billing
    if item.EndDate.IsZero() || item.EndDate.After(effectiveDate):
        item.EndDate = effectiveDate
        SubscriptionLineItemRepo.Update(ctx, item)  // direct repository call, NOT DeleteSubscriptionLineItem
```

**Entity type:** The filter must use `SubscriptionLineItemEntityTypePlan`. `SubscriptionLineItemEntityTypeSubscription` is used for TaxRate and AddonAssociation entity types elsewhere ŌĆö it is not used for subscription line items in this service and must not be included.

**Persistence mechanism:** The helper must call the repository layer **directly** (e.g., `SubscriptionLineItemRepo.Update()`), not via `DeleteSubscriptionLineItem()`. The `DeleteSubscriptionLineItem` service function contains a validation that rejects `effectiveFrom < item.StartDate`. Even though the pre-start guard above skips those items, using `DeleteSubscriptionLineItem` would introduce a dependency on that internal check and deviates from the intended data operation (we are updating `EndDate`, not deleting the record). Use a direct repository update.

**Guard conditions explained:**

- `item.StartDate.After(effectiveDate)` ŌĆö Pre-start cancellation: the subscription was cancelled before this line item's start date. The line item never became active. Setting `EndDate = effectiveDate` would create `EndDate < StartDate`, an inconsistent record. Skip these items; billing is protected by the subscription-level `EndDate` check (for immediate cancellations) and by Fix 3's per-item `EndDate` filter (for the scheduled cancellation window ŌĆö see Fix 3).
- `item.EndDate.IsZero() || item.EndDate.After(effectiveDate)` ŌĆö Only update items whose EndDate is either unset or extends beyond the cancellation point. Items already terminated before `effectiveDate` are left untouched.

**Called for both cancellation types:**

- **Immediate cancellation** ŌĆö `effectiveDate = now` (or provided date)
- **Scheduled cancellation** ŌĆö `effectiveDate = cancel_at` (future date); plan line items are terminated at schedule-creation time, not when the executor fires

**Scope boundary:** Only plan line items. Addon line items continue to be handled by the existing `cancelAddonsForSubscription()`.

### Scheduled Cancellation Timing

For scheduled cancellations (`CancellationType = end_of_period` or `scheduled_date`):

1. **At schedule creation time** (`CancelSubscription()` call): Fix 2 sets `item.EndDate = cancel_at` on all eligible plan line items. The subscription record gets `CancelAt = cancel_at` but `EndDate` remains nil at this point.
2. **When the executor fires** (billing cron detects `CancelAt <= now`): The executor sets `sub.EndDate = sub.CancelAt` and `sub.SubscriptionStatus = Cancelled`. It does not touch line items ŌĆö they are already correctly terminated from step 1.

Both timestamps align to `cancel_at`, so the state is consistent. The executor path does not need to call `cancelPlanLineItemsForSubscription`.

## Fix 3: Billing Respects Line Item `EndDate`

**File:** `internal/service/billing.go`
**Functions:** `FilterLineItemsToBeInvoiced()` and `CalculateFixedCharges()`

### Problem

`CalculateFixedCharges()` skips line items where `item.StartDate.After(periodEnd)` but does not skip items where `item.EndDate` has already passed. After Fix 2 ensures `EndDate` is always set on cancellation, this check is needed to enforce it at invoice time.

### Change

In both `FilterLineItemsToBeInvoiced()` and `CalculateFixedCharges()`, add an early-continue guard inside the iteration loop over line items:

```go
// Skip line items that ended before the billing period started
if !item.EndDate.IsZero() && !item.EndDate.After(periodStart) {
    continue
}
```

In `FilterLineItemsToBeInvoiced`, this guard goes inside the deduplication loop before any existing billing-period checks. In `CalculateFixedCharges`, it goes inside the line item iteration loop alongside the existing `item.StartDate.After(periodEnd)` skip.

### Behavior After Change

- A line item with `EndDate = March 31` is excluded from the April billing period
- Line items with `EndDate = zero` (no expiry) continue to be included normally
- Combines with Fix 2: Fix 2 sets `EndDate` on cancellation; Fix 3 enforces it in billing

### Billing Guard Coverage by Cancellation Type

| Cancellation type | Primary billing guard |
|---|---|
| Immediate | `sub.EndDate` (set immediately) ŌĆö subscription-level check in `FilterLineItemsToBeInvoiced` |
| Scheduled (pre-executor) | `item.EndDate` (set to `cancel_at` by Fix 2) ŌĆö per-item check added by Fix 3 |
| Scheduled (post-executor) | `sub.EndDate` (set by executor) + `item.EndDate` (already set) ŌĆö both guards active |
| Pre-start immediate cancel | `sub.EndDate` (set immediately) ŌĆö line items skipped by helper, subscription-level check covers billing |

For scheduled cancellations before the executor fires, `sub.EndDate` is nil. Fix 3's per-item `EndDate` filter is the **operative billing guard** during this window. The subscription-level `EndDate` check in `FilterLineItemsToBeInvoiced` only activates after the executor runs.

### Usage Charges ŌĆö No Change Needed

`CalculateUsageCharges()` does not filter line items by `EndDate` directly. It relies on the repository layer, which applies active-status filtering at the DB level only when **both** `ActiveFilter = true` and `CurrentPeriodStart` are set in `SubscriptionLineItemFilter` ŌĆö both are required for the filter to take effect. Usage charges for cancelled line items are additionally blocked upstream: events after cancellation are rejected by `isSubscriptionValidForEvent()` and `item.IsActive(event.Timestamp)` in the event post-processing service. No changes to the usage charge path are needed.

## Success Criteria

- A subscription with `start_date` in the future can be created via the API without error
- Events timestamped before `start_date` are rejected (existing behavior, unchanged)
- Cancelling a subscription sets `EndDate` on all eligible plan line items (not just addons)
- Cancelling a future-dated subscription before its `start_date` results in zero billing (line items with `StartDate > effectiveDate` are skipped by the helper; subscription-level `EndDate` protects billing)
- Billing does not invoice line items whose `EndDate` is before the billing period start
- No double billing across subscription boundaries
- No changes to existing subscriptions (forward-only)
