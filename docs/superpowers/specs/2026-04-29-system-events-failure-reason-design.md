# System Events: `failure_reason` Column

**Date:** 2026-04-29  
**Status:** Approved

## Problem

Webhook delivery failures in the `system_events` table are dropped silently. When Svix or native HTTP delivery fails, the error is logged to application logs but never persisted. The only way to investigate a failed webhook is to dig through logs. There is no durable, queryable record of why a delivery failed.

## Goal

Add a `failure_reason` TEXT column to `system_events`. Populate it at every failure point so the cause is always visible directly on the row. Never clear it — it persists even after a successful delivery, providing a permanent audit trail of the last known failure.

## Approach

Option A: single `failure_reason` column. Minimal scope, fits the existing data model, no new tables or joins required.

## Design

### 1. Schema — `ent/schema/systemevent.go`

Add one nullable optional field:

```go
field.String("failure_reason").Optional().Nillable()
```

- Type: nullable `TEXT` in PostgreSQL
- Go type: `*string` — `nil` when no failure has occurred
- Never cleared by `OnDelivered()` or any success path
- Overwritten on each subsequent failure

### 2. Migration

- New Ent migration generated via `make generate-migration`
- Raw SQL file added to `migrations/postgres/`:

```sql
ALTER TABLE system_events ADD COLUMN IF NOT EXISTS failure_reason TEXT;
```

### 3. Repository — `internal/repository/ent/systemevent.go`

New method `OnFailed(ctx context.Context, id, reason string) error`:

- Targeted UPDATE: sets `failure_reason = reason` where `id = id`
- No conditions on current state — always overwrites
- No other fields touched

Domain interface in `internal/domain/systemevent/` updated with the same method signature.

`OnDelivered()` is **not changed** — it only stamps `webhook_message_id` and `published_at`.

### 4. Failure Write Points

**Primary — `absorbDeliveryError()` in `internal/webhook/handler/handler.go`:**
- After logging the error, call `repo.OnFailed(ctx, event.ID, err.Error())`
- Error from `OnFailed` is logged but does not change the return value
- Return behavior (nil to Watermill) stays unchanged — ack behavior is not affected

**Secondary — stale retry activity in `internal/temporal/activities/cron/webhook_outbound_retry_activities.go`:**
- After a retry attempt fails (currently only logged), also call `repo.OnFailed(ctx, event.ID, err.Error())`
- Same pattern: log + persist, control flow unchanged

### 5. What Does NOT Change

- Return values / Watermill ack behavior
- `OnDelivered()` success path
- Retry scheduling / Temporal workflow structure
- Kafka message handling logic
- Any other system_events fields

## Failure Reason Behavior

| Scenario | `failure_reason` |
|---|---|
| Never attempted | `NULL` |
| First delivery fails | Set to error message |
| Retry also fails | Overwritten with latest error |
| Eventually delivered successfully | Kept as-is (last failure preserved) |
| Delivered on first attempt | `NULL` (never written) |
